#                                                                                                              ** AI工业“医生”诊断报错项目**

---

## 任务背景

* ### 1. 行业痛点与挑战

  在现代智能制造领域，数控机床、自动化生产线等工业设备每秒都会产生海量的传感器日志。然而，传统的故障排查面临以下三大困境：

  - **非直观性**：原始日志（如 `ERR_001: temp_high_98c`）由代码和缩写组成，一线操作员难以快速理解其真实含义。
  - **知识断层**：设备维修高度依赖资深工程师的个人经验。当专家不在场时，普通员工很难从厚重的技术手册或历史工单中快速检索到解决方案。
  - **时效性差**：传统的人工查阅手册模式在紧急停机事故中反应过慢，每分钟的停机都会造成巨大的生产损失。

  ### 2. 任务目标

  本项目旨在开发一套 **“AI 工业医生”智能诊断平台**，利用大语言模型（LLM）的自然语言处理能力，将冰冷的机器代码转化为人类可读的诊断建议。

  - **核心功能**：实现原始日志的自动解析、技术文档的智能检索（RAG）以及多专家协同会诊逻辑。
  - **交付标准**：系统需具备高度集成的可视化界面，并能生成可供归档的专业技术诊断报告（PDF）。

  ### 3. 系统技术定位

  本系统定位于 **工业物联网 (IIoT) 的决策辅助层**。通过接入 DeepSeek API 和本地向量数据库（FAISS），系统能够闭环完成从“发现异常”到“检索方案”再到“生成报告”的全流程自动化。

## 系统架构

系统架构如下：

![](http://tc38t9k2i.sabkt.gdipper.com/AI工厂智能医生（2）.png)

**系统的四层闭环运作机制**

1.**用户交互与输入层**：用户通过 Streamlit 界面 上传原始机床日志。前端实时显示系统状态，并提供趋势分析和 PDF 报告导出入口。

2.**数据储存层（知识底座）**：系统将历史故障案例 与工程师编写的 `.txt` 技术文档 存入 `data/` 目录，并通过 `knowledge_engine.py` 转化为 **FAISS 向量索引**，为诊断提供专业背景。

3.**业务处理层（中枢调度）**：`processor.py` 作为核心调度器，根据 `config.py` 的全局配置，同时开启两条逻辑流：

- **诊断流**：调用单或多专家协同模式，对日志进行多维度解析。
- **检索流 (RAG)**：从向量库中匹配最相关的技术指导建议。

4.**外部依赖与反馈层**：系统将日志与检索到的知识整合成 Prompt 发送给 **DeepSeek API**，获取最终生成的专家诊断方案，并以流式文本形式实时返回给前端界面。 

## 前端页面介绍

前端页面展示：

![](http://tc38t9k2i.sabkt.gdipper.com/前端首页.png)

**本项目主要包含了四大功能**

**1.故障诊断**

首先便是主要的核心功能，系统的诊断功能，在页面中用户可以上传所需检测的日志文件，之后再点击下面的立即开始诊断便可以通过已知的知识库（RAG增强），进行分析，将机器故障语言传入deepseek大模型，再将翻译为人类能理解的语言返回输出流，输出故障状态，故障原因，解决方法，为了方便开始的使用，我们加了样本范例，点击右侧“载入演示用例”再点击开始诊断，便可以看到该演示用例的诊断结果，为了更加详细的诊断，我们还加入了多专家诊断模式，多方面来分析理解，给出更详尽的结果。

![](http://tc38t9k2i.sabkt.gdipper.com/故障诊断.png)

**2.工程师知识库**

在这里你可以上传自家工程师总结的经验以及实例，构建独特的针对性的知识库，这能够进一步增强，诊断时的知识索引能力，保证获得更加好的有用的结果。

![](http://tc38t9k2i.sabkt.gdipper.com/工程师知识库.png)

**3.经典案例库**

我们考虑到，有时候分析诊断结果时，如果觉得该条诊断结果很典型或者是很有价值，那么我们可以通过收藏该条诊断结果案例，进入到经典案例库当中，便于之后需要的时候重新查看，当然了如果觉得没用了也可以删除该收藏案例。

![](http://tc38t9k2i.sabkt.gdipper.com/收藏案例库.png)

**4.运行状况趋势**

为了方便一段时间的问题观察，我们还有运行状况趋势观看，这里总结了近段时间的错误故障代码，以及故障状态的观看，能够方便一段时间的测试下，总结一系列的机器故障问题，可以生成出趋势分析报告，方便后续工程师进一步优化和修复机器故障毛病。

![](http://tc38t9k2i.sabkt.gdipper.com/运行状态趋势.png)

**5.除此之外**

为了优化使用，我们在界面上显示了各个状态，像API是否连接上，诊断条数有多少，知识库建立状态，以及是否开启会诊模式，为了方便回顾，我们还有历史记录功能，能够记住以往诊断的案例与结果，为了方便现实使用，我们对于诊断报告，可以用保存出来pdf用于进一步分析。

## 开发总结

###  1.任务达成概况

本项目已全面完成“入门任务”要求的各项指标。系统不仅实现了基础的**工业日志自动化翻译与诊断**，更通过 **RAG (检索增强生成)** 技术解决了大模型在特定工业领域知识匮乏的问题。

### 2. 核心功能实现

- **智能诊断引擎**：基于 DeepSeek API，实现了“单/多专家协同会诊”模式，能够从机械、电气等多个专业维度拆解故障。
- **企业知识库集成**：利用 `knowledge_engine.py` 与 **FAISS 向量数据库**，实现了对本地技术手册的深度检索，确保诊断建议具备专业参考依据。
- **工业化交互界面**：采用 Streamlit 构建了深色系工业风 UI，集成了实时系统监控、故障趋势分析及 PDF 诊断报告导出功能。

### 3. 技术架构亮点

- **高内聚低耦合**：系统采用分层架构设计（展示层、处理层、数据层、依赖层），通过 `config.py` 实现集中化配置。
- **闭环进化机制**：建立了“用户反馈/案例收藏流”，允许系统在运行过程中不断积累经典故障案例，实现知识资产的持续沉淀。
- **工程化交付规范**：项目配套了完整的 Visio 架构图、Markdown 文档以及由 PicGo + 七牛云托管的图床系统，符合专业软件开发交付标准。

### 4. 自我评价与展望

通过本项目，我深入实践了 **LLM Agent** 调度逻辑与 **RAG 架构** 的工业化落地。

- **已解决难点**：成功攻克了 PDF 中文渲染乱码、多模型异步流式输出以及异构数据向量化等技术问题。
- **后续优化**：未来可考虑接入更多实时传感器数据流，实现从“事后诊断”向“事前预警”的预测性维护转型。
